STEP 1 / 6

準備・リポジトリのセットアップ

まずリーダー(A)がGitHubにリポジトリを作り、メンバー全員がクローンします。

👑
リーダー A
GitHubでリポジトリ作成・ファイルをpush・招待を送る
💻
メンバー B
招待を承認してリポジトリをクローンする
🎨
メンバー C
招待を承認してリポジトリをクローンする

(1) リーダーA：GitHubにpush

BASH
# ダウンロードしたgit-learningフォルダで実行
cd git-learning
git init
git add .
git commit -m "feat: 初期セットアップ"
git remote add origin https://github.com/[あなたのID]/git-learning.git
git push -u origin main

(2) B・C：クローンして初期設定

BASH
git clone https://github.com/[リーダーのID]/git-learning.git
cd git-learning
git config user.name "自分の名前"
git config user.email "メールアドレス"
💡
GitHubの Settings → Collaborators からメンバーを招待できます。招待されたメンバーはメールかGitHub通知で承認してください。
STEP 2 / 6

ブランチ作成・編集・push

各メンバーが自分のブランチを作り、index.html に自分のカードを追加します。mainには直接pushしません。

main *-------------------------------------> \ \ feature/member-B *--*--* (B が作業中) \ feature/member-C *--* (C が作業中)

メンバーBの場合

BASH
# ブランチを作成して切り替え
git checkout -b feature/member-B

# index.html を編集（メンバーカードを追加）
# ↓ このHTMLを「TODO: メンバーBが...」のコメントの下に追加
HTML
<div class="card">
  <div class="card-avatar">B</div>
  <h3>メンバーB</h3>
  <p class="role">フロントエンド担当</p>
  <p class="bio">自己紹介を書こう</p>
  <div class="skills-tags">
    <span>CSS</span><span>JavaScript</span>
  </div>
</div>
BASH
git add index.html
git commit -m "feat: メンバーBのカードを追加"
git push origin feature/member-B
💡
メンバーCも同様に feature/member-C ブランチで作業します。アバターは「C」、色は変えてみましょう。
STEP 3 / 6

プルリクエスト・レビュー

pushしたブランチをGitHub上でPRにして、リーダーにレビューしてもらいます。

誰が何をする場所
B / C GitHub上でPRを作成（base: main ← compare: feature/自分） GitHub Web
A（リーダー） コードを確認してコメントを入れる（"Changes requested"） GitHub Web
B / C コメントに対応して追加コミットをpush ターミナル
A（リーダー） "Approve" して "Merge pull request" GitHub Web
💡
レビューのポイント： typo・書き方の統一・コンフリクットマーカーが残っていないか確認しましょう。

レビュー後の修正コミット方法

BASH
# ファイルを修正後
git add index.html
git commit -m "fix: レビュー指摘を修正"
git push origin feature/member-B
# → PRに自動的に追加される
STEP 4 / 6

コンフリクトを意図的に発生させて解消する

BとCが同じ行を同時に編集することで、コンフリクトを体験します。これが一番重要な練習です！

⚡
シナリオ： Bは <title> を「Git学習 | チームB」に変更。Cは同じ行を「Git練習 | チームC」に変更。Bのマージ後にCをマージしようとするとコンフリクト発生！

コンフリクト発生時の画面

<<<<<<< HEAD (main) <title>Git学習 | チームB</title> ======= <title>Git練習 | チームC</title> >>>>>>> feature/member-C

解消手順

BASH
# 1. コンフリクトファイルを確認
git status

# 2. エディタでファイルを開き、マーカーを見つける
#    (<<<<<<< / ======= / >>>>>>>)

# 3. 話し合って1つに決め、マーカーを削除
#    例：<title>Git学習 | チームで習得</title>

# 4. 解消後にコミット
git add index.html
git commit -m "fix: タイトルのコンフリクトを解消"
⚠️
コンフリクト解消後のファイルに <<<<<<< や >>>>>>> が残っていないか必ず確認しましょう！

練習できるコンフリクトパターン

ファイル場所難易度
index.html<title>タグ易
index.htmlメンバーカードの順番中
css/style.css--color-accent の値中
js/main.jsアニメーション速度難
STEP 5 / 6

rebase と merge の違いを体験

同じ結果でも履歴の形が変わります。git log --graph で視覚的に確認しましょう。

mergerebase
履歴の形ひし形（分岐が残る）一直線
マージコミット自動で作られる作られない
使いどきチームの共有ブランチ自分の作業ブランチ整理
force push不要必要になる場合あり

merge の場合

BASH
git checkout main
git merge feature/member-B
# → "Merge branch 'feature/member-B'" コミットが作られる
git log --oneline --graph --all

rebase の場合

BASH
git checkout feature/member-C
git rebase main
# → CのコミットがmainのHEADの後ろに"移植"される
git log --oneline --graph --all
# → 履歴が一直線になっているのを確認！
⚠️
rebase後にpushするときは git push --force-with-lease が必要な場合があります。チームで共有しているブランチへのrebaseは避けましょう。
